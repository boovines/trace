## Codebase Patterns
- Cross-module contract consistency lives in `synthesizer.check_contracts` (wrapper: `scripts/check_contracts.sh`). It layers over `synthesizer.check_fixtures`: (1) JSON-Schema validity of every `contracts/*.schema.json` via `Draft202012Validator.check_schema` + explicit `$schema` dialect pin, (2) trajectory-fixture validity via `TrajectoryReader(traj_dir)` (reuses the reader's line-numbered error messages), (3) skill-fixture validity via `check_all_fixtures()` (schema + round-trip + markdown/meta cross-check), (4) `meta.trajectory_id → metadata.json.id` linkage inside `_check_trajectory_link`. Both scripts share the `CheckResult` API (`failures: list[str]`, `ok` property, `fail(path, reason)`) — downstream gate scripts should either extend this helper or mirror the same shape so failure messages stay uniform and LLM-actionable.
- `httpx.ASGITransport` does NOT stream response bodies — it buffers the full response before any bytes reach the client, so SSE endpoints served via sse-starlette appear to hang forever when consumed via `ac.stream("GET", ...)` + `aiter_lines`/`aiter_bytes` in tests. Test the generator directly (`api_module._sse_event_generator(session, fake_request)`) with a `_FakeRequest` stub exposing `is_disconnected()`; verify the route wiring separately (pin `SSE_PING_INTERVAL_SECONDS == 15` and walk `router.routes` to confirm the stream path exists). Full ASGI SSE integration belongs in a uvicorn-backed smoke test, not the unit suite.
- `_sse_event_generator` emits 3 events per state transition when the session is already past the first transition (state_change, draft_ready, question_ready all in ONE poll iteration). Tests that iterate the generator should `break` at the expected event count and then `request.disconnect()` so the next poll returns cleanly. The generator never exits on its own while the session is still in a non-terminal state — disconnect is the only clean termination hook aside from the session reaching COMPLETED/ERRORED/CANCELLED.
- Background tasks from `asyncio.create_task` must hold a strong reference to avoid GC mid-flight. `api.py` uses a module-level `_BACKGROUND_TASKS: set[asyncio.Task[None]]` plus a `add_done_callback(_BACKGROUND_TASKS.discard)` — downstream async stories (S-015 e2e) should use the same `_schedule_background` helper rather than calling `create_task` directly.
- uv workspace: dev extras on workspace members aren't picked up by `uv sync --extra dev` at the root; use `uv sync --all-extras --all-packages` (or run per-package) to install a workspace member's dev deps.
- Synthesizer package uses **src layout** (`services/synthesizer/src/synthesizer/`) with hatchling; declare `[tool.hatch.build.targets.wheel] packages = ["src/synthesizer"]` so the wheel finds it.
- `pytest` exits 5 on "no tests collected" — that's NOT zero. If a story's AC requires `pytest ... exits 0`, add at least one smoke test to the new suite.
- All synthesizer tests run with `TRACE_LLM_FAKE_MODE=1` + `ANTHROPIC_API_KEY=test-dummy-key` forced via an autouse fixture in `tests/synthesizer/conftest.py`, plus the `anthropic_mock` respx router is autouse-installed on every test so an unmocked call to `api.anthropic.com` raises instead of hitting the network.
- Per-service `pyproject.toml` declares its own `[tool.mypy] strict = true` so `uv run mypy --strict services/synthesizer` uses service-scoped config (not the root's). Root `[tool.mypy]` still covers `services` + `gateway` for repo-wide checks.
- `contracts/*.schema.json` is located at runtime by walking up from `Path(__file__)` until a directory containing `contracts/` is found; this keeps modules working from both the editable workspace install and a future published wheel without hard-coding a relative path.
- jsonschema's `FormatChecker` validates `uuid` out-of-the-box, but `date-time` requires the optional `rfc3339-validator` dep — schemas relying on strict date-time enforcement should combine `type: "string"` with a regex pattern or a post-validation Python check, not just the `format` keyword.
- `jsonschema` ships no PEP 561 type stubs; use `# type: ignore[import-untyped]` on the import line to keep `mypy --strict` green without pulling in `types-jsonschema`.
- When two branches of one function both assign `sorted(set_diff)` to the same variable name, mypy pins the variable's element type from the first assignment. Use distinct names per branch (e.g., `only_md_steps` vs `only_md_params`) to avoid spurious `incompatible type` errors.
- SKILL.md format (locked by S-003): sections in strict order `# Title` → description → `## Parameters` → `## Preconditions` → `## Steps` → `## Expected outcome` → optional `## Notes`. Parameter bullet shape: `` - `name` (type, required|optional[, default: value]) — description ``. Defaults: strings double-quoted, integers bare, booleans `true`/`false`. Empty bullet sections render `_None._`. Destructive steps prefixed with `⚠️ [DESTRUCTIVE]` exactly.
- Pydantic v2 `model_config = ConfigDict(frozen=True)` on `ParsedSkill`/`Step`/`Parameter` makes round-trip tests use bare `==` for equality and also prevents accidental mutation by downstream stages.
- The parameter-reference regex `(?<!\\)(?<!\{)\{NAME\}(?!\})` plus pre-stripping fenced/inline code is the approved pattern for finding `{name}` refs — mirrors S-002 schema cross-check but is stricter (handles escaped `\{`, `{{double}}`, inline backticks). Use `extract_parameter_refs` from `skill_doc` rather than re-rolling the regex in future stories.
- The trajectory schema (`contracts/trajectory.schema.json`) requires every event to carry `seq` (integer ≥ 1, unique per trajectory), and supports `click` and `app_switch` event kinds plus an optional `target: {label, role, bundle_id}` sub-object on clicks. Downstream stories (S-005 preprocessor, S-008 destructive matcher) read click `target.label` — populate it on every click event in fixtures.
- Reference-fixture trajectories live at `fixtures/trajectories/<slug>/` with `metadata.json`, `events.jsonl`, and `screenshots/0001.png` (a 67-byte 1×1 grayscale PNG). Tests that mutate a fixture MUST `shutil.copytree` it into `tmp_path` first — the committed fixtures are ground truth for S-016 and other downstream tests.
- `TrajectoryReader` validates `metadata.json` against `definitions.metadata` and each `events.jsonl` line against `definitions.event` using two separate `Draft202012Validator` instances, cached via `lru_cache`. Error messages always include the line number AND the event's `seq` so a malformed fixture is trivial to locate.
- Event objects use `ConfigDict(extra='allow', frozen=True)` — same pattern as `ParsedSkill`. Unknown keys flow through for forward-compat, and frozen events are safe to cache / hash / compare in downstream stages.
- Events are sorted by `seq` on load, so `iter_events()` / `iter_events_by_type()` always yield in logical order regardless of on-disk line order. Duplicate `seq` values raise on construction (would make `get_screenshot_path` ambiguous).
- Trajectory digest (`synthesizer.preprocess.DigestEntry`) is the single shape fed to the draft-generation prompt. `seq=0` on a digest entry marks a synthetic idle gap (never a real event since the schema's seq minimum is 1); `scroll_run` entries reuse the first collapsed scroll's seq. Downstream stories (S-007 draft prompt, S-012 revision) should treat `seq=0`/`kind in {scroll_run, idle}` as synthetic and not try to trace them back to a single event.
- `synthesizer.preprocess.APP_NAMES` maps common macOS bundle_ids to human names (Chrome, Finder, Notes, Slack, ...). Unknown bundle_ids fall through to the bundle_id itself — extend the map here rather than adding per-call branches in prompt code.
- The preprocessor drops `mouse_move`/`mouse_down`/`mouse_up` as noise but keeps every `click`, `text_input`, and `app_switch` exactly once. When constructing synthetic trajectories in tests, omit mouse_move/down/up entirely unless you specifically want to exercise the noise filter.
- The synthesizer never imports the Anthropic SDK directly — all calls go through `synthesizer.llm_client.LLMClient`. That wrapper owns retries (the SDK's `max_retries=0`), fake mode, and cost logging in one place. Downstream stories (S-007 draft, S-012 revision, S-014 similarity) MUST construct one `LLMClient` per session and pass it down rather than instantiating per call — keeps the cost log coherent and makes the no-stray-network assertion meaningful.
- Trace data dir resolution is `synthesizer.llm_client.trace_data_dir()`: honors `TRACE_DATA_DIR` env override (used by tests with `tmp_path`), then falls back to `TRACE_PROFILE` selecting `Trace` (prod) vs `Trace-dev` (dev), defaulting to prod. New code that writes anywhere under `~/Library/Application Support/Trace[-dev]/` should call this helper rather than re-deriving the path.
- Pricing for cost estimates lives in `PRICING_USD_PER_MTOK` in `llm_client.py` — single source of truth. Add new model entries here when bumping; the test `test_pricing_table_has_required_models` enforces presence of Sonnet 4.5 + Haiku 4.5 entries.
- Fake-mode canned responses are keyed by `compute_request_hash(system, messages, model, max_tokens, tools)` (sha256 over canonical JSON). Tests register them with `save_fake_response(...)` against a per-test `TRACE_FAKE_RESPONSES_DIR` so they don't leak across tests; downstream story integration tests that need committed fixtures live under `fixtures/llm_responses/`.
- `costs.jsonl` line shape is locked: `{timestamp_iso, module, model, input_tokens, output_tokens, cost_estimate_usd, context_label}`. S-018 (budget guardrails) reads this back, so don't drop or rename keys.
- `LLMResponse.cost_estimate_usd` is the per-call figure *as billed* — always `0.0` in fake mode, matched 1:1 to the `costs.jsonl` line for that call. Downstream code that accumulates cost across a session (S-007 DraftResult, S-012 revision, S-018 budget check) should sum this field rather than re-estimating from `input_tokens` + model, because a re-estimate would double-bill fake-mode tests at Sonnet prices.
- Draft-generation user message shape: a single `user` content block list containing exactly one `type="text"` digest block followed by up to 20 `type="image"` base64 PNG blocks. `synthesizer.draft.build_user_content(preprocessed, reader)` is the single builder — call it from tests that need to compute request hashes, never hand-roll the blocks.
- Draft retry chain: on any validation failure (`_ResponseValidationError`), `generate_draft` appends the bad response as an `assistant` turn and the feedback string as the next `user` turn, then re-calls. Hash stability for canned responses relies on the feedback strings being pure functions of the bad response — if S-012 reuses this retry shape, keep `_parse_response_json`/`_validate_markdown`/`_validate_meta_and_cross` as the single source of feedback text.
- `fixtures/prompt_examples/*.txt` holds the few-shot examples embedded in `DRAFT_SYSTEM_PROMPT` at import time; files are read in sorted filename order for determinism. Changing the examples silently invalidates every canned fake-mode response hash — regenerate them in lockstep or commit both together.
- `[tool.ruff.lint.per-file-ignores]` at the repo root now has an E501 exemption for `draft_prompt.py` — long prose inside the embedded LLM system prompt. When adding new LLM-prompt modules, add them to the same per-file-ignores rather than hand-wrapping paragraphs (hard wraps degrade prompt readability for the model).
- `synthesizer.destructive_matcher.apply_destructive_matcher` uses **label-in-step-text binding**, not positional index-to-click pairing: for each click with a destructive label, find the earliest un-bound step whose text case-insensitively contains the label substring. Positional pairing is wrong because step lists include non-click entries (app_switch, text_input) that misalign the index with `iter_events_by_type("click")`. The digest's click step summaries (``Clicked button labeled "Send" in Chrome``) embed the label, so substring match finds the right step reliably. Downstream: if the draft prompt ever stops including the literal label in step text, the matcher silently stops flagging — keep the label in the summary.
- `synthesizer.slug`: canonical slug shape is `^[a-z][a-z0-9_]{2,39}$` (constants `SLUG_PATTERN`, `SLUG_REGEX`, `SLUG_MIN_LEN=3`, `SLUG_MAX_LEN=40`). `slugify(name)` pipeline: NFKD transliterate → lowercase → collapse any `[^a-z0-9_]+` run into single `_` → strip leading non-letters (digits, underscores, leftover junk) → rstrip `_` → truncate to 40, rstrip `_` again → error if <3 chars. `resolve_unique_slug(base, dir)` only uses numeric suffixes (`base_2`, `base_3`, ...) — the `_v2`/`_copy`/`_new` naming ambiguity is disallowed by design. `validate_user_slug(input, dir) -> (bool, reason)` returns human-readable rejection messages suitable for surfacing to the user in the Q&A UI. Downstream (S-010 writer, S-011 session approve, S-013 API `/approve` endpoint) should always call `resolve_unique_slug` before a disk write and prefer `validate_user_slug` before accepting a user-provided override.
- `synthesizer.writer.SkillWriter.write(parsed, meta, reader, skills_root, *, cost_total_usd=0.0) -> WrittenSkill` is the ONLY allowed path for creating files under `skills/` or touching `index.db`. Sequence: (1) validate round-trip (`parse(render(parsed)) == parsed`) + `validate_meta` + `validate_meta_against_markdown` + slug regex; (2) check slug-not-in-dir AND slug-not-in-index — both must be clear, caller is responsible for calling `resolve_unique_slug` first; (3) create `skills_root/<slug>/` with mode 0o700; (4) atomic write `SKILL.md.tmp` → `fsync` → rename, then same for `skill.meta.json.tmp`; (5) copy up to 5 preview screenshots into `preview/` (renumbered `01.png`..`05.png`); (6) commit row to SQLite `index.db`. Any failure before or during commit rolls back the entire skill directory via `shutil.rmtree(..., ignore_errors=True)` — the invariant "skill is either fully registered or fully absent" is enforced, not hoped for.
- `synthesizer.writer.index_db_path()` returns `trace_data_dir() / "index.db"` — shared with the recorder per the PRD. Schema: `CREATE TABLE IF NOT EXISTS skills (slug TEXT PRIMARY KEY, name TEXT NOT NULL, trajectory_id TEXT NOT NULL, created_at TEXT NOT NULL, step_count INTEGER NOT NULL, destructive_step_count INTEGER NOT NULL)`. `_ensure_schema(conn)` is idempotent (IF NOT EXISTS); downstream stories (S-013 GET /skills, DELETE /skills) should reuse this helper rather than inlining CREATE TABLE.
- Preview screenshot selection (`synthesizer.writer._pick_preview_indices(n, cap)`): when `n <= cap`, returns `[0..n-1]`; otherwise spread across `[0, n-1]` using `round(i * (n-1) / (cap-1))` for `i in range(cap)` and dedupe while preserving order. Python's `round` uses banker's rounding so `n=6, cap=5` → `[0, 1, 2, 4, 5]` and `n=100, cap=5` → `[0, 25, 50, 74, 99]` — tests pin these exact values. Screenshots are copied (not symlinked) via `shutil.copy2`; the AC is explicit so the skill dir is self-contained even if the trajectory is later deleted.
- Atomic text write (`synthesizer.writer.SkillWriter._atomic_write_text`): `open(".tmp")` → `write` → `flush` → `os.fsync` → `os.rename`. The `os.fsync` is imported as `synthesizer.writer.os.fsync` for monkeypatch reachability in crash-simulation tests — reference it as `"synthesizer.writer.os.fsync"` in `monkeypatch.setattr` rather than patching the global `os` module.
- `synthesizer.session.SynthesisSession` is the state-machine orchestrator: states are a `StrEnum` (`generating_draft`, `awaiting_answer`, `generating_revision`, `awaiting_approval`, `saving`, `completed`, `cancelled`, `errored`); transitions go through `_transition(target)` which consults the declarative `_LEGAL_TRANSITIONS` dict and raises `IllegalStateTransition` on any illegal move. The constructor is intentionally non-blocking (no thread spawn) — `start_draft()` is explicit so the HTTP layer (S-013) can wrap it in `asyncio.to_thread` while unit tests run it inline. `draft_fn` and `revise_fn` are injected callables (Protocols `DraftFn` / `ReviseFn`) so S-012 can plug in `generate_revision` without a circular dep, and tests can script behaviour deterministically.
- Session persistence: every successful draft + revision writes `<trace_data_dir>/synthesis_sessions/<session_id>/draft.json`; `cancel()` and `approve() → COMPLETED` both `shutil.rmtree` the dir. Persistence happens AFTER the state transition so `draft.json` always reflects the CURRENT resting state (`awaiting_answer`/`awaiting_approval`), never an intermediate (`generating_draft`). v1 does NOT rehydrate sessions from disk on service restart — documented PRD limitation.
- Session approval slug handling: when the caller passes `slug=None`, the LLM-suggested slug in `draft.meta["slug"]` is piped through `resolve_unique_slug` (silent numeric suffix on collision); when the caller passes an explicit `slug="..."`, it goes through `validate_user_slug` and a collision raises `SlugError` rather than renaming. Matches the PRD's user-vs-LLM asymmetry: users who type a slug expect to see their exact input or an error, not a silent rename.
- `SessionStore` is the single in-memory registry of live sessions. `evict_stale()` is the sweep function — it prunes BOTH terminal (completed/cancelled/errored) sessions AND non-terminal sessions older than `ttl_seconds` (default 30 min). Clock is injectable (`clock: Callable[[], float]`) so tests advance time without `time.sleep`. The async cleanup loop itself is deferred to S-013 (`asyncio.create_task` wrapping `evict_stale` on an interval); `cancel_cleanup()` / `cleanup_cancelled()` let the API layer shut down cleanly.
- Session tests never spin up real LLM: `_fake_draft`/`_fake_revise` callables return pre-built `DraftResult` objects. The pattern `session = SynthesisSession(draft_fn=lambda preprocessed, client, *, reader: draft, ...)` plus `session.start_draft()` is the canonical way to drive a session through each state deterministically. `DraftGenerationError(message, attempts=[], last_error=...)` requires keyword args `attempts` and `last_error` — raising it with positional args trips the `__init__` signature.
- `synthesizer.revise.generate_revision(*, current_draft, question, answer, client, reader, model=..., max_tokens=..., context_label=...)` is the S-012 phase-2 entry point. Mirrors `generate_draft`'s retry shape (JSON parse / markdown parse / meta validation → corrective user turn, capped at `MAX_REVISION_LLM_CALLS=3`), reuses the exact private validators (`_parse_response_json`, `_validate_full_response`, `_ResponseValidationError`) from `synthesizer.draft` so hash-stability of fake-mode retry tests is preserved across phases. Destructive matcher (S-008) is re-applied after every successful response — paranoia but cheap. Empty/whitespace `answer` raises `ValueError` BEFORE any LLM call. Cost & call counts are cumulative: returned `DraftResult.total_cost_usd = current_draft.total_cost_usd + sum(attempt.cost_estimate_usd)`; same additive pattern for `llm_calls`.
- Revision user-message shape: a single `type="text"` block containing four labelled sections — `CURRENT_MARKDOWN` (fenced), `CURRENT_META` (pretty-printed JSON), `ANSWERED_QUESTION` (JSON `{id, category, text, answer}`), `REMAINING_QUESTIONS` (JSON array, with the just-answered question filtered out). No screenshots — the revision call operates on the current draft text, not the raw trajectory. Tests that compute request hashes should call `synthesizer.revise.build_revision_user_content(current_draft, question, answer)` to build the exact block structure.
- `REVISE_SYSTEM_PROMPT` lives in `synthesizer.revise_prompt` and is separate from `DRAFT_SYSTEM_PROMPT` — draft retains few-shot digest examples, revision omits them (the "few shots" at revision time ARE the current markdown/meta, which is always in-context). Both prompts require the same `{markdown, meta, questions}` output shape so the shared validation helpers apply unchanged. `pyproject.toml` now has an E501 ignore for `revise_prompt.py` matching the pattern already used for `draft_prompt.py`.
- `synthesizer.similarity.score_skill_similarity(generated, golden, client, *, model=SIMILARITY_MODEL, max_tokens=DEFAULT_SIMILARITY_MAX_TOKENS, context_label="synthesizer:similarity") -> SimilarityScore` is the S-014 snapshot-similarity entry point. Uses `SIMILARITY_MODEL = "claude-haiku-4-5"` (5x cheaper than Sonnet; pricing entry already present in `PRICING_USD_PER_MTOK`). Two dimensions (`step_coverage`, `overall`) come from a single Haiku call; two dimensions (`parameter_match`, `destructive_match`) are computed LOCALLY as binary 0.0/1.0 via frozenset equality. Parameter signature uses `(name, type, required)` triple — defaults & descriptions intentionally ignored to avoid wording-drift false positives. Destructive signature is the set of destructive step numbers. Strict JSON parsing: no retry on a malformed rubric response — raises `SimilarityScoringError`; the design choice (PRD rationale) is that a malformed rubric is a prompt/model issue worth surfacing, not masking.
- Similarity rubric prompt (`SIMILARITY_SYSTEM_PROMPT` in `synthesizer.similarity`) instructs Haiku to return `{step_coverage, overall, reasoning}` — NO `parameter_match`/`destructive_match` (those are caller-computed). Keeping the model out of binary-fact dimensions is both faster and removes a non-determinism source. The `reasoning` string ends up in `smoke_report.json` (S-017) so humans skimming a real-mode run can see why a fixture scored what it did.
- S-014 conftest adds a `similarity_scorer` pytest fixture (lazy import of `score_skill_similarity`) so S-017 can depend-inject the scorer; `Callable[..., Any]` return type on the fixture passes mypy strict without coupling conftest to the similarity module at collection time.
- E2E fake-mode session tests (S-015) drive the real `SynthesisSession(draft_fn=generate_draft, revise_fn=generate_revision)` against canned fake responses. Pattern: register draft response at `hash(DRAFT_SYSTEM_PROMPT, [user: build_user_content])`, then for each revision build the expected `DraftResult` manually (`_build_draft_result_from_canned`) and register the next revision response at `hash(REVISE_SYSTEM_PROMPT, [user: build_revision_user_content(prior_draft, q, answer)])`. The destructive matcher runs inside `generate_draft`/`generate_revision`, so if the canned markdown already has the correct `⚠️ [DESTRUCTIVE]` markers the matcher is a no-op and `DraftResult.markdown` equals the canned markdown byte-for-byte — this is what makes the chained request-hash construction tractable.
- `_discover_destructive_steps(slug, preprocessed, reader)` pattern: build a skeleton SKILL.md with zero destructive flags, parse it, run `apply_destructive_matcher` on the skeleton, and read back which step numbers were flagged. This is the canonical way to pre-compute expected destructive step numbers without duplicating keyword-matching logic in test code. For the 5 reference fixtures only `gmail_reply` step 4 (Send) is destructive; other fixtures have no destructive clicks in their trajectories.
- Golden skill fixtures (`fixtures/skills/<slug>/SKILL.md` + `skill.meta.json`) are HAND-CRAFTED ground truth for the similarity test harness (S-014/S-017). Ralph iterations MUST NOT regenerate them from synthesized drafts — that collapses the snapshot comparison into a tautology. `scripts/check_fixtures.sh` (backed by `synthesizer.check_fixtures.check_all_fixtures`) validates every committed pair: schema, round-trip, markdown/meta cross-check, and trajectory_id → `fixtures/trajectories/<dir>/metadata.json.id` linkage. Wire this into any new quality-gate script alongside `ruff`/`mypy`/`pytest`.
- Fixture trajectory_id linkage is BY ID, not by slug: `_check_trajectory_link` walks every `fixtures/trajectories/<dir>/metadata.json` looking for a matching `.id` field. Slug renames don't break the link — the UUID in `meta.trajectory_id` is the stable reference. Any future cross-fixture validator (S-019 check_contracts.sh) should scan by id rather than by directory name.
- `validate_meta_against_markdown` scans the ENTIRE markdown for `{name}` refs (description, preconditions, steps, expected outcome, notes — not just the Steps section). Inline-backticked `` `name` `` in the Parameters bullet section is correctly ignored by `_INLINE_CODE_RE`, so documenting a parameter doesn't double-count as a reference. When hand-crafting fixtures or writing drafts that reference a parameter in prose (Expected outcome, Preconditions), the parameter MUST appear in `meta.parameters`.
- Opt-in / gated tests (e.g., S-017 `test_real_smoke.py`) combine module-level `pytestmark = pytest.mark.skipif(not os.environ.get("ENV_VAR"), ...)` with module-local fixture overrides that shadow conftest autouse fixtures by name. This is the canonical way to let one test bypass the suite-wide hermetic guards (`_force_fake_mode`, `_no_stray_anthropic_calls`, `anthropic_mock`) without touching `conftest.py` and without risking the gated code path executing during a normal `pytest tests/synthesizer/` run.

## 2026-04-23 - S-011 Synthesis session state machine
- Added `services/synthesizer/src/synthesizer/session.py` (~420 lines) exposing `SynthesisSession`, `SessionStore`, `SessionState` (StrEnum), `IllegalStateTransition`, `DraftFn` / `ReviseFn` Protocols, `DEFAULT_SESSION_TTL_SECONDS=1800`, `session_dir()`, `sessions_root()`, plus the `_LEGAL_TRANSITIONS` transition table and `TERMINAL_STATES` frozenset.
- `SynthesisSession` constructor is cheap: allocates a session_id (uuid4 hex), stores reader/client/writer references, sets state=GENERATING_DRAFT. No threading — `start_draft()` is explicit so tests can exercise the illegal-transition case (answer_question while state=generating_draft) without racing a background thread.
- State-machine core: `_transition(target)` consults `_LEGAL_TRANSITIONS` and raises `IllegalStateTransition` on illegal moves. Legal flow: GENERATING_DRAFT → AWAITING_ANSWER (or AWAITING_APPROVAL if no questions) → GENERATING_REVISION → AWAITING_ANSWER/AWAITING_APPROVAL → SAVING → COMPLETED. Any non-terminal state can transition to CANCELLED or ERRORED. Terminal states have no legal outgoing transitions (approve-after-completion, cancel-after-completion both raise).
- Draft phase: `start_draft()` calls `preprocess_trajectory(reader)` then `draft_fn(preprocessed, client, *, reader)` matching `generate_draft`'s signature. Catches `DraftGenerationError` and unexpected exceptions, both route to `_fail(msg)` which sets `error` and transitions to ERRORED. On success, stores the draft, seeds `questions` from `draft.questions`, sets `costs_so_far_usd = draft.total_cost_usd`, transitions to the appropriate next state, then calls `_persist_draft()`.
- Q&A phase: `answer_question(question_id, text)` validates state=AWAITING_ANSWER, non-empty text (raises ValueError on blank), known question_id (raises ValueError on unknown). Transitions to GENERATING_REVISION, calls `revise_fn(current_draft=..., question=..., answer=..., client=..., reader=...)`, catches errors into ERRORED. On success, appends to `answered`, updates draft/questions/costs, then transitions to AWAITING_ANSWER if `next_question() is not None` else AWAITING_APPROVAL. `next_question()` returns the first question whose id isn't in `answered`.
- Approve phase: `approve(slug=None)` validates state=AWAITING_APPROVAL, transitions to SAVING, resolves slug (`resolve_unique_slug` for None, `validate_user_slug` for user-provided), calls `SkillWriter.write(...)`, transitions to COMPLETED and cleans up persistence. `SlugError` from user-provided collision transitions to ERRORED and re-raises so the API layer can return 409. Any other write failure also transitions to ERRORED and re-raises.
- Cancel: `cancel()` transitions any non-terminal state → CANCELLED, cleans up `synthesis_sessions/<sid>/` via `shutil.rmtree(..., ignore_errors=True)`. Calling cancel on a terminal state raises `IllegalStateTransition`.
- Persistence: `_persist_draft()` writes `<trace_data_dir>/synthesis_sessions/<session_id>/draft.json` with {session_id, trajectory_id, state, markdown, meta, questions, answered, costs_so_far_usd}. Called AFTER state transitions so the file reflects the resting state, not an intermediate. `snapshot()` returns a UI-facing dict with the same shape plus `next_question` and `error`.
- `SessionStore`: in-memory dict keyed by session_id. `add`/`get`/`remove`/`__contains__`/`__len__`/`ids()` + `evict_stale()` which prunes both terminal sessions and non-terminal sessions older than `ttl_seconds` (default 1800s = 30min). Injectable `clock: Callable[[], float]` for fake-time tests. `cancel_cleanup()`/`cleanup_cancelled()` are hooks for the S-013 async cleanup loop.
- Added `tests/synthesizer/test_session.py` with 23 tests: full happy-path (3 questions → approve → WrittenSkill), no-questions draft skips straight to AWAITING_APPROVAL, illegal transitions (answer during generating_draft, approve before draft, start_draft after completion, cancel from terminal), cancel from each non-terminal state (generating_draft, awaiting_answer with persistence cleanup, awaiting_approval with no skill written), draft + revision error paths, approve with colliding LLM slug (renames to `_2`), approve with colliding user slug (raises SlugError + ERRORED state), approve with custom user slug (honored verbatim), empty/whitespace answer raises ValueError without changing state, unknown question_id raises, draft persistence file written + updated after revision, SessionStore add/get/remove/duplicate-rejection, evict_stale past TTL with fake clock, evict_stale prunes terminal sessions, cleanup_cancelled flag, snapshot shape progression with state.
- Files changed: services/synthesizer/src/synthesizer/session.py (new), tests/synthesizer/test_session.py (new), scripts/ralph/prd.json (passes: true for S-011), scripts/ralph/progress.txt (this entry + 5 patterns).

- **Learnings for future iterations:**
  - `ruff UP042` flags `class Foo(str, Enum)` and suggests `class Foo(StrEnum)` — Python 3.11's `enum.StrEnum` is cleaner and members still behave like strings (so `"generating_draft" == SessionState.GENERATING_DRAFT` holds). Use StrEnum for any new string-valued enum in this codebase.
  - Protocol types (`DraftFn`, `ReviseFn`) are the right way to express "callable matching this signature" under `mypy --strict`. The draft function's positional-then-keyword signature (`def __call__(self, preprocessed, client, *, reader)`) matches `generate_draft` exactly; keeping it as a Protocol rather than `Callable[..., DraftResult]` gives better type-checking at call sites.

- Added `services/synthesizer/src/synthesizer/slug.py` (~190 lines) exposing `slugify`, `resolve_unique_slug`, `validate_user_slug`, `SlugError`, plus `SLUG_PATTERN` / `SLUG_REGEX` / `SLUG_MIN_LEN` / `SLUG_MAX_LEN` constants. No new runtime deps — uses stdlib `re` + `unicodedata` only.
- `slugify` uses NFKD normalization + ASCII encode-ignore for best-effort transliteration (Émilie → emilie, Café → cafe, Résumé → resume). Emoji and non-Latin scripts are dropped entirely — English-only by design per CLAUDE.md. Leading digits/underscores are stripped so `"123 Notes Daily"` → `"notes_daily"` and `"_foo"` → `"foo"`. Trailing underscores are stripped before and after the 40-char truncation so truncating mid-run never leaves a dangling separator.
- `resolve_unique_slug` detects collisions by directory (or file) presence at `skills_dir/<candidate>`; a stray regular file sitting where a skill dir would go also blocks the slug (overwriting random files is worse than choosing another slug). Raises `SlugError` if the numeric suffix would push the candidate past `SLUG_MAX_LEN` — unreachable for sensible bases but worth being explicit.
- `validate_user_slug` returns `(True, trimmed_slug)` or `(False, human_reason)`. Reason strings name the specific defect (too short, too long, uppercase, starts-with-digit, disallowed chars with the offenders listed, already-exists). UI can surface these verbatim.
- Added `tests/synthesizer/test_slug.py` with 58 tests: 21 parametrized happy-path slugify cases (punctuation, accents, leading digits, emoji, multiple separators), 9 SlugError cases (empty, only-digits, only-underscores, <3 chars), truncation + trailing-underscore-strip after truncation, macOS directory-safety for 6 problematic inputs, 7 `resolve_unique_slug` tests (base-free, dir-missing, 1 collision, 5 collisions → _6, file collision, invalid base, max-length overflow), 15 `validate_user_slug` tests covering every rejection path and the min/max length boundaries.
- Files changed: services/synthesizer/src/synthesizer/slug.py (new), tests/synthesizer/test_slug.py (new), scripts/ralph/prd.json (passes: true for S-009), scripts/ralph/progress.txt (this entry + the slug pattern).
- Quality gates: `uv run ruff check services/synthesizer tests/synthesizer` ✓, `uv run mypy --strict services/synthesizer` ✓ (12 source files), `uv run pytest tests/synthesizer/` ✓ (245 passed, +58 from S-008).
- **Learnings for future iterations:**
  - `unicodedata.normalize("NFKD", s).encode("ascii", "ignore").decode("ascii")` is the stdlib pattern for transliteration — no Unidecode dep needed. It splits combining diacritics off their base letters, then the ASCII-ignore encode drops the marks (and every non-ASCII codepoint, including emoji, CJK, etc.). Downstream code that needs transliteration should reuse this idiom — it's about 5 lines, not worth a helper.
  - `str.rstrip("_")` after truncation is mandatory — slicing `s[:40]` can leave a trailing `_` if position 40 happened to be a separator, and the resulting slug would fail `SLUG_REGEX`. The slugify tests hit this boundary explicitly (`"a"*39 + " " + "b"*10`).
  - The pattern `^[a-z][a-z0-9_]{2,39}$` bounds TOTAL length to 3–40 (first char + 2–39 more). A common off-by-one would be to think "2 to 39 more" means total 2–39. The tests at the min/max boundaries (`"abc"` and `"a" + "b"*39`) exist to pin this down.
  - `SlugError` subclasses `ValueError` so callers that catch `ValueError` broadly still see it; keep it as a specific subclass so new call sites can tighten up.
  - Ruff's `RUF059` flags unused unpacked tuple members (`ok, reason = ...` when only `ok` is asserted). The fix is to either prefix with `_` or actually use the variable; the test suite now asserts on the reason text wherever it's unpacked, which doubles as a sanity check on the user-facing message quality.
  - Ruff's `RUF002` flags `–` (EN DASH) and `—` (EM DASH) in docstrings as ambiguous with hyphen-minus. Prose in code docstrings should use ASCII `-` or switch to EM DASH only where clearly a sentence break; raw EN DASH in ranges like "3–40" must be rewritten as "3-40". (The synthesizer package already has EM DASH uses in existing docstrings that ruff has been accepting — only EN DASH and the multiplication `×` glyph trip the checker. The E501 allowance on `draft_prompt.py` does NOT help here.)

- Added `services/synthesizer/src/synthesizer/budget.py` (~300 lines) exposing `BudgetConfig`, `BudgetMonitor`, `BudgetStatus` (StrEnum: OK/WARNING/EXCEEDED), `BudgetCheckResult`, `load_budget_config`, `budget_config_path`, plus `DEFAULT_SESSION_CAP_USD=1.00` / `DEFAULT_DAILY_CAP_USD=5.00` / `WARNING_THRESHOLD=0.80` constants. Two caps: per-session (cumulative draft + revisions) and daily (sum of today's `module='synthesizer'` entries in `costs.jsonl`). Defaults kick in when `<trace_data_dir>/config.json` is absent; invalid values raise `ConfigurationError` so operators notice before a live session silently skips the guardrail.
- Monitor deduplication: per-session warnings fire once per `session_id`, daily warning fires once per monitor lifetime. Both write to stderr directly (not just LOGGER.warning) so the line always reaches the user console regardless of logging config. Daily sum parses `timestamp_iso` via `datetime.fromisoformat`, filters to `entry.get('module') == 'synthesizer'`, and skips malformed/partial lines rather than failing the whole check.
- Integration with `SynthesisSession` (S-011): added `budget_monitor: BudgetMonitor | None` kwarg and a new `_enforce_budget()` helper that runs after both `start_draft()` and `answer_question()`. When `check_session_cost` returns EXCEEDED, `_fail("per-session cost cap exceeded")` transitions to ERRORED and the method short-circuits before the next state transition. Monitor-less sessions (the default in fake-mode e2e tests) bypass the check entirely — preserves existing test contracts.
- Integration with `POST /synthesize/start` (S-013): added `_BUDGET_MONITOR_FACTORY: Callable[[], BudgetMonitor] = BudgetMonitor` module-level hook. The endpoint instantiates a fresh monitor per request, runs `check_daily_cost()`, and returns HTTP 429 with `detail="daily synthesizer cost cap of $X.XX exceeded (current $Y.YYYY); try again tomorrow"` before touching the trajectory directory or creating a session. The accepted monitor is passed into the `SynthesisSession` constructor so per-session enforcement applies for the session's lifetime. Tests swap `_BUDGET_MONITOR_FACTORY` to inject a clock-fixed, cost-file-pinned monitor.
- Added `tests/synthesizer/test_budget.py` (~690 lines, 32 tests): config loading (defaults / disk / partial override / 7-parametrized invalid types / non-object / malformed JSON), per-session classify + warning dedup + per-session-scoped dedup, daily classify + sum + multi-session accumulation + warning + malformed-line resilience, 4 `SynthesisSession` integration tests (draft exceeds → ERRORED, under cap → AWAITING_APPROVAL, revision trip → ERRORED, no monitor → unaffected), and 2 HTTP endpoint tests (429 when daily cap exceeded, 200 when under cap).
- Files changed: services/synthesizer/src/synthesizer/budget.py (new), services/synthesizer/src/synthesizer/session.py (budget_monitor kwarg + _enforce_budget helper, +22 lines), services/synthesizer/src/synthesizer/api.py (daily check at /synthesize/start + pass monitor into session, +13 lines), tests/synthesizer/test_budget.py (new), scripts/ralph/prd.json (passes: true for S-018), scripts/ralph/progress.txt (this entry).
- Quality gates: `uv run ruff check services/synthesizer tests/synthesizer` ✓, `uv run mypy --strict services/synthesizer` ✓ (19 source files, +1 from S-017), `uv run pytest tests/synthesizer/` ✓ (425 passed, 1 skipped — +32 from S-017), `./scripts/check_fixtures.sh` ✓.
- **Learnings for future iterations:**
  - `bool` is a subclass of `int` in Python, so a naive `isinstance(x, int | float)` check lets `True`/`False` through as valid cap values. The `_validate_cap` helper explicitly rejects booleans first — the parametrized invalid-type test includes `True` to pin this down. Any future numeric config validators in this codebase should do the same.
  - Module-level factory hooks (`_BUDGET_MONITOR_FACTORY: Callable[[], BudgetMonitor] = BudgetMonitor`) are the established pattern for DI in `api.py` — matches the existing `_DRAFT_FN`/`_REVISE_FN`/`_STORE` naming. Tests inject via `monkeypatch.setattr(api_module, "_BUDGET_MONITOR_FACTORY", _factory)`. Do not pass monitors through request state or FastAPI Depends — the module-level swap keeps the endpoint signature clean and matches the existing conventions.
  - `SynthesisSession`'s `_enforce_budget()` is called after `costs_so_far_usd` is updated, BEFORE the next state transition. Calling it before the cost update would race (costs hasn't accumulated yet); calling after the transition would let the state leak forward one step past the cap. This order is load-bearing — if future stories add new cost-accruing state transitions (e.g., a separate "polish" phase), insert `_enforce_budget()` calls at the same post-update / pre-transition point.
  - The daily-cost check in `/synthesize/start` runs BEFORE `trajectory_dir.is_dir()` — a client hitting a not-yet-existing trajectory with the daily cap exceeded gets a 429, not a 404. This is deliberate: the cap is an ambient constraint on the system, while the 404 is conditional on the caller's input. Error priority matters for UI behavior (a 429 should suggest "try tomorrow", a 404 should suggest "record first").
  - The monitor writes warnings to stderr via bare `print(..., file=sys.stderr)` AND `LOGGER.warning(...)`. This belt-and-suspenders pattern ensures the warning reaches the user even when logging is configured above WARNING level (common in daemon contexts where `logging.basicConfig(level=ERROR)` is set to quiet noisy deps). Reusable for any future guardrail where "user sees this" matters more than "logs capture this".
