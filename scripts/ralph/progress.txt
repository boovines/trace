## Codebase Patterns
- Runner uses a **src layout** (`services/runner/src/runner/`). Tests import `from runner.xxx`, resolved via `pythonpath = ["services/runner/src"]` in the root `pyproject.toml` pytest config. Do NOT add a nested `services/runner/runner/` package; that was the old flat layout and has been removed.
- The root pyproject owns the shared `ruff`, `mypy`, and `pytest` config. The runner's own `pyproject.toml` only adds a mypy `files = ["src"]` override so `uv run mypy --strict services/runner` targets the runner package directly.
- Safety gate: every live-mode code path must call `runner.safety.require_live_mode()` in `__init__`. The gate is strict string equality against `TRACE_ALLOW_LIVE="1"` — `"true"`, `"yes"`, `"0"`, etc. all disable live mode. `tests/runner/conftest.py` autouses a fixture that force-unsets the flag before every test; opt in with the `live_mode_allowed` fixture.
- Tests mock Anthropic via `respx` (see `anthropic_mock` fixture) — never hit the real API.
- Gateway (`gateway/main.py`) imports the runner as `from runner.api import router` (installed-package style via the pytest pythonpath). Synthesizer/recorder are still on the flat layout and import via `from services.{name}.{name}.api` — those packages will switch to src layout when their respective branches reach X-001-equivalent stories.
- JSON schemas in `contracts/` are the on-disk contract; the runner pairs each with a pydantic model in `runner.schema` (e.g. `RunMetadata`) for typed in-memory use. Both must agree — `extra="forbid"` on the model mirrors `additionalProperties: false` on the schema. Round-tripping uses `model_dump(mode="json", exclude_unset=True)` so unset optional fields do not leak back as nulls.
- Pydantic v2 serializes UTC datetimes with a trailing `Z` (not `+00:00`) via `mode="json"`. Tests that assert round-trip identity must store the canonical `Z` form in fixtures.
- `types-jsonschema` is in the shared dev extras so mypy --strict accepts `jsonschema` imports without module-level overrides.
- Shared contracts that need validation beyond JSON Schema (e.g. list-length, lowercase) live at the repo root under `contracts/` and are policed by `scripts/check_contracts.sh`. Runtime code loads them through a module in `runner.*` with `@lru_cache(maxsize=1)`; the schema path is computed from `Path(__file__).resolve().parents[4]` (for `services/runner/src/runner/*.py`).
- Non-ASCII literals in tests (fullwidth letters, NBSP) should be written as `\uXXXX` escape sequences, not pasted as glyphs — ruff's RUF001 flags ambiguous characters and breaks lint otherwise.
- The synthesizer package is now importable at the top level as `synthesizer.*` (not `services.synthesizer.synthesizer.*`). `services/synthesizer` is on both pytest `pythonpath` and mypy `mypy_path`. The recorder is still flat-layout and reached as `services.recorder.recorder.*` until its own X-001-equivalent story lands. Gateway uses `from synthesizer.api import ...` to match.
- Cross-module typechecking: leaving `services/synthesizer/synthesizer` out of mypy `files` while keeping it on `mypy_path` avoids the "source file found twice" ambiguity. Synthesizer code still gets strict-checked because the runner imports it and mypy follows those imports.
- SKILL.md format (locked by X-004 parser): a `# Title` line, a `## Steps` section whose body is `N. text` lines (1-based, contiguous). Destructive steps begin with `⚠️ ` (fully qualified) or `⚠ ` (bare). The parser strips the marker from `.text` and sets `destructive=True`. Prose between title and `## Steps` is ignored but preserved on disk.
- Skill meta/markdown semantic check lives in `synthesizer.schema.validate_meta_against_markdown` — it enforces that `meta.destructive_steps` is exactly the set of `⚠️`-marked step numbers (symmetric: either-direction drift is an error).
- Parameter substitution: `meta.parameters[*]` is the declaration. A parameter is required iff it has NO `default` key at all OR `default` is `null`. Unknown parameters from the caller raise `UnknownParameterError`. Escaped braces `\{` / `\}` survive substitution as literal `{` / `}`. Non-string defaults (e.g. integer `30`) are stringified via `str()` before substitution.
- Atomic metadata writes: `runner.run_writer._atomic_write_bytes` uses `os.open` + `os.write` + `os.fsync` + `os.replace` with a `.tmp` sibling, then removes the tmp on any exception. Always patch `runner.run_writer.os.fsync` (not `os.fsync`) in tests so the module-local reference is intercepted. `update_status` re-runs pydantic validation via `model_dump() | updates → model_validate` — `model_copy(update=...)` does NOT re-validate and will happily accept a bogus status enum.
- Retina geometry: `runner.coords` exposes `points_to_pixels`/`pixels_to_points` (factor-of-`scale_factor`) and a combined `resized_pixels_to_points(x, y, mapping)` where `mapping.scale_from_resized_to_points = (original_w / resized_w) / scale_factor` — use `ImageMapping` to convert Claude-space coordinates back to display points in one multiply. `capture_and_normalize` is pure normalization; the `source_image=None` default raises `NotImplementedError` because live capture is X-009's `LiveScreenSource`. Use `DryRunDisplayInfo` (2880x1800 / 1440x900 / 2.0) as the canonical dry-run display. `round(x)` returns `int` directly — never wrap in `int(round(...))` (triggers RUF046).

Do NOT re-export it from `input_adapter.py` — that created a circular import (`live_input` needs `MouseButton`/`ScrollDirection` literals from `input_adapter`). Callers import directly: `from runner.live_input import LiveInputAdapter`.
- `pytest_addoption` must live at the **test-tree root** (`tests/conftest.py`), not in a subdirectory conftest — pytest only discovers plugin hooks from rootdir conftests at startup. The `--run-live-input` flag gates `@pytest.mark.live_input` tests via a `pytest_collection_modifyitems` hook that adds a skip marker unless the flag is set.
- For PyObjC-backed modules like `runner.live_input`: import the C constants (e.g. `kCGEventLeftMouseDown`) and functions (`CGEventPost`) at module top level. Tests patch them via `monkeypatch.setattr(live_input_module, "CGEventPost", MagicMock())` — the module-local reference is what actually gets called, so patching the Quartz-global won't intercept. `int(kCGEvent*)` coerces PyObjC numeric constants to Python ints when building flag bitmasks with `|`.
- Quartz scroll events: `CGEventCreateScrollWheelEvent(source, units, wheelCount, *wheels)` takes a variadic wheel-value tuple. For vertical scroll use `wheelCount=1, wheels=(delta_y,)`; for horizontal use `wheelCount=2, wheels=(0, delta_x)`. mypy --strict wants the heterogeneous tuple typed as `tuple[int, ...]` before the branching assignment.
- `LiveScreenSource` lives in `runner.live_screen`; Quartz/AppKit symbols (`CGWindowListCreateImage`, `CGPreflightScreenCaptureAccess`, `CGRectInfinite`, `NSBitmapImageRep`, `NSBitmapImageFileTypePNG`) are imported at module top level so tests can `monkeypatch.setattr(live_screen_module, "...", MagicMock())` against the module-local binding. `_cg_image_to_pil` is an instance method rather than a module function so tests can override it via `monkeypatch.setattr(LiveScreenSource, "_cg_image_to_pil", ...)`. Preflight (`CGPreflightScreenCaptureAccess`) is only consulted when capture returns None — on the happy path we never pay for it. Untyped PyObjC returns from `CGWindowListCreateImage` need an explicit `image: object | None = ...` annotation before returning, otherwise `mypy --strict` flags a no-any-return warning.
- Agent runtime split: `runner.agent_runtime` defines the `AgentRuntime` Protocol + `AgentResponse` dataclass (pure abstraction, no SDK coupling). `runner.claude_runtime` is the default implementation (`ClaudeRuntime`) plus an `OpenAIRuntime` stub. Fake mode is env-driven (`TRACE_LLM_FAKE_MODE=1`) and matches a `fixtures/llm_responses/runner_<slug>.json` script by `marker` substring in `system_prompt`. Turn selection is by `run_turn` call count (1-based), NOT input hash — simpler, deterministic, and tests don't have to maintain hash constants. Cost logging goes to a caller-provided `costs_path`, not a global; tests pass `tmp_path`. The SDK's built-in retry is disabled (`max_retries=0`) so our exponential-backoff policy (1,2,4,8,16s × 5 attempts) is the only retry layer; patch `runner.claude_runtime.asyncio.sleep` in tests to skip the real waits. The `computer_20250124` tool's `display_width_px`/`display_height_px` come from `current_image_mapping.resized_pixels` when set, else from `DryRunDisplayInfo` downscaled to longest-edge 1568 (= 1568×980) — call `runtime.set_image_mapping(...)` after each screenshot.
- Anthropic SDK exception taxonomy: `AuthenticationError` subclasses `APIStatusError`, so `except AuthenticationError` MUST come before `except APIStatusError` in the `try` block — the reverse order would swallow 401s into the retry branch. `RateLimitError` (429) also subclasses `APIStatusError`; catch via `APIStatusError` and branch on `.status_code`.
- Anthropic `Message.content` is a list of typed block objects (TextBlock, ToolUseBlock, …). Convert to plain dicts via `[block.model_dump() for block in msg.content]` before handing off to downstream parsers — keeps `AgentResponse.content_blocks` JSON-serializable and matches the fake-mode fixture shape one-for-one.
- Execution prompt (`runner.execution_prompt`) is load-bearing for safety and must stay deterministic. Protocol strings (`DESTRUCTIVE_PROTOCOL`, `COMPLETION_PROTOCOL`, `ERROR_PROTOCOL`, `VERIFICATION_PROTOCOL`, `DRY_RUN_NOTICE`) are module-level `Final` constants so downstream parsers (X-012) and tests pin them by reference — never copy-paste the strings. The literal tag `<needs_confirmation step="N"/>` is the single source of truth; the harness-layer keyword matcher (X-015) is the belt-and-suspenders backup.
- Rendering a `ParsedSkill` back to markdown: the parser strips the `⚠️ ` prefix and exposes `step.destructive` separately. When re-rendering for the prompt, re-attach an explicit `⚠️ [DESTRUCTIVE] ` prefix so the rendered text matches the `DESTRUCTIVE_PROTOCOL` instruction wording verbatim.
- Dispatcher (`runner.dispatcher`) translates a `ToolCallAction` into `InputAdapter` + `ScreenSource` calls and returns a `ToolResult(content_blocks, new_image_mapping, is_error)`. Every side-effecting action (click/type/key/scroll/wait) ends with a fresh screenshot; `mouse_move` is the sole exception. `wait` duration is capped at `MAX_WAIT_SECONDS=10.0`. Coordinates arrive in resized-pixel space and are mapped to display points via `resized_pixels_to_points(..., image_mapping)` inside the dispatcher — callers pass the `image_mapping` from the most recent screenshot. `double_click` is implemented as two `click()` calls on the same point (InputAdapter has no click-count param). Click `text` field (e.g. `"cmd+shift"`) is parsed via `parse_key_expression` into a modifier list. The dispatcher never raises on malformed tool_input — bad `action`, missing/negative/out-of-bounds coordinates, non-string `text`, wrong `scroll_direction`, negative `scroll_amount`/`duration`, unknown action name all return `ToolResult(is_error=True, content_blocks=[{type:"text", text:"..."}])` WITHOUT calling the adapter. Booleans are explicitly rejected in numeric fields (coordinate, scroll_amount, duration) even though `bool` subclasses `int`.
- Tool-call parser (`runner.parser`) is the first destructive gate at the text-tag level: `parse_agent_response(AgentResponse) -> ParsedAction` returns a tagged union of `ToolCallAction | ConfirmationRequest | WorkflowComplete | WorkflowFailed | UnknownAction`. Priority order in same turn: `ConfirmationRequest` > `WorkflowComplete` > `WorkflowFailed` > `ToolCallAction` > `UnknownAction`. A confirmation tag ALWAYS drops any concurrent `tool_use` block (safety). A malformed confirmation (missing quotes, extra attrs, step <= 0, non-integer) also drops the tool_use — we detect the attempt via a loose `<\s*needs_confirmation\b` pattern and return `UnknownAction` rather than silently falling through. Parser never raises. Content blocks are duck-typed as dicts (X-010 converts via `model_dump()`); real SDK block objects are not supported.
- Run budgets (`runner.budget`): `RunBudget` (frozen dataclass, five caps, defaults 500k/50k in-out tokens, 600s wall clock, 30 actions/min, 100 total) + `BudgetTracker` (mutable). Hard limits (tokens/time/total) return `BudgetStatus.budget_exceeded(reason)` → executor aborts; soft per-minute cap returns `rate_limited(wait_seconds)` → executor sleeps. `time_source` is injectable (defaults to `time.monotonic`); tests pass a `FakeClock` callable instead of monkeypatching. Sliding window is a `deque[float]` trimmed in `check()` with `oldest <= cutoff` (60s old → expired). Per-skill overrides load via `RunBudget.from_skill_meta(meta)` reading optional `meta["runtime_limits"]`; unknown nested keys are rejected both python-side AND by `skill-meta.schema.json`'s `additionalProperties: false` on the nested object. `"runtime_limits": null` must no-op to defaults, not error. Bool-rejection (`isinstance(x, bool)` before the numeric check) guards every int-typed input.
- Executor (`runner.executor`) is the X-017 orchestration boundary: per turn → budget check → `agent_runtime.run_turn` → accumulate tokens/cost → parse → dispatch per action kind. Finalisers (`_finalize_succeeded|failed|aborted|budget_exceeded|agent_stuck`) are the only paths that set terminal status; every exit goes through `RunWriter.update_status` so on-disk metadata stays schema-valid. Image mapping starts at a `_default_image_mapping()` seeded from `DryRunDisplayInfo` downscaled to 1568 longest edge (matches `ClaudeRuntime._default_display_dims()` exactly) and updates every time the dispatcher returns an `image` content block. Fourth consecutive `UnknownAction` trips `agent_stuck` with `error_message=AGENT_STUCK_REASON` so a model narrating without acting cannot burn tokens forever. Dry-run and execute share the same executor — only the prompt mode and the gate's dry-run short-circuit differ.
- KillSwitch (`runner.kill_switch`): per-run `asyncio.Event` registry shared across the service. Executor calls `register(run_id) -> asyncio.Event` on start and `unregister(run_id)` in a `finally` block; API layer / hotkey handler calls `kill(run_id, reason)`. `kill()` is idempotent (repeat calls return False), no-op before register, no-op after unregister — all three are logged at INFO. Process-global singleton is exposed via `get_global_kill_switch()` for the API layer; tests construct their own `KillSwitch()` for isolation. The Executor owns the kill/confirmation race: `_await_decision_or_kill` races `queue.await_decision` against the kill event, synthesizes an abort decision on kill, and ALSO calls `queue.submit_decision(run_id, abort)` so the pending-state bookkeeping cleans up. `_run_turn_cancellable` races the LLM call against the kill event and `.cancel()`s the turn task on kill — that's what cancels the in-flight httpx request. A kill check also sits just before `dispatch_tool_call` so no new adapter calls happen after kill (relevant for the safety invariant that PyObjC calls must not fire after kill). Default abort_reason constant `KILL_SWITCH_REASON = "kill_switch"` lives in `runner.executor`.
- Fake-mode fixtures in `fixtures/llm_responses/runner_*.json` must use click coordinates `x < 1440` and `y < 900` — the `TrajectoryScreenSource`'s blank canvas is 1440×900, and `dispatcher._parse_coordinate` silently returns an error result on out-of-bounds coords (the click is never forwarded to the adapter). Tests that only assert on run status won't catch this; tests that inspect `DryRunInputAdapter.get_recorded_calls()` will.
- Trajectory-event annotations use `note="step=N;target=<label>[;destructive=true]"`. `synthesizer.trajectory_reader._parse_note` is the single parser; extend it rather than inventing a new annotation format. The reader filters events to `mouse_down` + `text_input` only (drops mouse_up/mouse_move/screenshot).
- ConfirmationQueue (`runner.confirmation`): backed by `dict[run_id, asyncio.Queue[ConfirmationDecision]]` with `maxsize=1`. `push_request(*, run_id, step_number, step_text, screenshot_ref, destructive_reason) -> ConfirmationRequest` raises `IllegalState` on double-push; `await_decision(run_id, timeout_seconds=300.0)` synthesizes `ConfirmationDecision(action='abort', reason='user_timeout')` on timeout and cleans up the pending + channel dicts in `finally`; `submit_decision(run_id, decision) -> bool` warning-logs and returns False when there is no pending request (legitimate race with timeouts / kill_run); `kill_run(run_id, reason='kill_switch')` wraps `submit_decision` with a synthetic abort for X-018. The queue's `ConfirmationRequest` dataclass shares its name with `runner.parser.ConfirmationRequest` on purpose — the parser one is the agent's tag-level signal, the queue one carries the full UI prompt (step_number+step_text+screenshot_ref+destructive_reason); callers needing both alias on import. WS shapes: `make_request_message(request, screenshot_url=...)` → `{type:'confirmation_request', run_id, step_number, step_text, destructive_reason, screenshot_url}`; `parse_response_message(payload) -> (run_id, ConfirmationDecision)` validates every field and raises `ValueError` on drift. Use builtin `TimeoutError` not `asyncio.TimeoutError` (ruff UP041).

  - `ParsedAction` as a PEP 604 tagged union (`ToolCallAction | ConfirmationRequest | ...`) keeps the executor's dispatch a simple `isinstance` chain. ## 2026-04-23 - X-018 - Kill switch implementation with 2-second abort guarantee
- Added `services/runner/src/runner/kill_switch.py` — `KillSwitch` class backed by `dict[str, _RunState]` where each `_RunState` carries an `asyncio.Event`, a reason, and a `killed` flag. Public surface: `register(run_id) -> asyncio.Event`, `unregister`, `kill(run_id, reason) -> bool` (idempotent), `is_killed`, `reason`, `is_registered`. Also exposes a process-global `get_global_kill_switch()` singleton for the API layer.
- Integrated into `Executor`: new `kill_switch: KillSwitch | None = None` constructor arg. On `run()` the executor registers, then wraps `_run_body()` in `try/finally` to always `unregister` + `writer.close()` on exit. Three kill checks are added: (1) `_kill_triggered()` at the top of every loop iteration, (2) `_run_turn_cancellable()` races the LLM call against the kill event and `task.cancel()`s the turn on kill — this is what cancels the in-flight httpx request, (3) a kill check immediately before every `dispatch_tool_call` so no new adapter calls happen after kill.
- Confirmation flows now use `_await_decision_or_kill()` which races `queue.await_decision` against the kill event; on kill, an abort decision is both synthesized and submitted back into the queue so the underlying task cleans up properly.
- `runner/api.py` now has `POST /run/{run_id}/abort` which calls `get_global_kill_switch().kill(run_id, reason='user_abort')`. Returns `{run_id, killed}` as 200 regardless — unknown / finished runs return `killed=False`.

- Added `tests/runner/test_kill_switch.py` — 20 async tests covering: KillSwitch unit behavior (register/unregister/kill/reason/idempotent), kill during slow turn aborts in <2s, kill during pending confirmation, kill before start is no-op, kill after finish is no-op, idempotent kill during run, no adapter calls after kill (turn 2 click blocked), default kill reason, regression test that `kill_switch=None` preserves existing behavior, httpx cancellation via patched `AsyncAnthropic.messages.create` sleep-forever coroutine that surfaces `CancelledError`, and API endpoint coverage via `fastapi.testclient.TestClient`.
- Quality gates: `uv run ruff check services/runner tests/runner` clean; `uv run mypy --strict services/runner` clean (23 source files); `uv run pytest tests/runner/` → 515 passed + 3 skipped (20 new); `scripts/check_contracts.sh` ok.
- Set X-018 `passes: true` in `scripts/ralph/prd.json`.
- **Learnings for future iterations:**
  - The `asyncio.wait([task_a, task_b], return_when=FIRST_COMPLETED)` pattern is the clean way to race two awaitables: inspect `done` for which won, cancel the loser via `.cancel()`, then `with contextlib.suppress(CancelledError, Exception): await loser_task` to drain it without crashing. `contextlib.suppress` is ruff SIM105-preferred over try/except/pass.
  - Cancelling an `asyncio.Task` that's awaiting `AsyncAnthropic.messages.create` surfaces as `asyncio.CancelledError` inside the SDK coroutine, which in turn cancels the underlying `httpx` socket read — that is the 2-second abort path. Tests don't need real respx delayed responses; patching `runtime._client.messages.create` with a `sleep_forever` coroutine that logs `CancelledError` is both cheaper and more reliable than respx async side_effects, which have historically been flaky.
  - Kill-during-confirmation works via a race in `_await_decision_or_kill`: on kill, the executor *also* calls `confirmation_queue.submit_decision(run_id, abort)` so the underlying `await_decision` task resolves and drops its pending-state bookkeeping. Don't rely on `ConfirmationQueue.kill_run` from inside `KillSwitch.kill` — that couples the two subsystems and forces the switch to know about the queue; the executor is already orchestrating both and is the right place for the race.
  - The "no further tool calls after kill" test requires a *slow second turn* (per-turn delay list `[0, 5, 0]`), not a uniform delay on all turns — with uniform 0.1s the kill fires AFTER turn 2 has already parsed and dispatched. Per-turn delays make the race deterministic: kill fires while turn 2 is sleeping in the SDK, the task gets cancelled, dispatch never runs.
  - FastAPI `TestClient` calls from `httpx` under the hood and works fine inside `@pytest.mark.asyncio`-marked async tests even though it's a sync client — no need to use `AsyncClient` unless the handler itself is async and we want to exercise the async runtime.
  - For the process-global KillSwitch singleton: tests must construct their own `KillSwitch()` for isolation. Only the API endpoint test actually pokes `get_global_kill_switch()` — and even that uses a randomized `run_id` and `unregister` in a `finally` block so cross-test leakage is impossible.

- **Learnings for future iterations:**
  - `fastapi.testclient.TestClient` uses `httpx` under the hood, and `httpx.Client` with `ASGITransport` spins up a NEW event loop per HTTP request and tears it down on return. That means any `asyncio.create_task()` scheduled from inside a POST handler — e.g. our original `start_run` — is cancelled as soon as the POST returns. Production uvicorn has one long-lived loop so the problem is invisible there, but it breaks any integration test that mixes POST and WS. Fix: the `RunManager` owns a `_BackgroundLoop` (an `asyncio.new_event_loop()` running forever on its own daemon thread) and submits run coroutines via `asyncio.run_coroutine_threadsafe`, then `asyncio.wrap_future(future, loop=asyncio.get_running_loop())` to give callers an awaitable bound to the caller's loop.
  - `asyncio.Queue`, `asyncio.Event`, and `asyncio.Future` are NOT thread-safe. `Event.set()` calls `Future.set_result()` which calls `loop.call_soon(...)` — that is only valid from the owning loop. For cross-thread delivery use `loop.call_soon_threadsafe(fn)`. The dispatch helper `_dispatch_on(loop, fn)` in `runner.kill_switch` is the reusable pattern — check `asyncio.get_running_loop()` against the stored loop; call `fn()` directly when they match, otherwise `loop.call_soon_threadsafe(fn)`. Pick this shape over always-threadsafe because tests that `kill(); assert event.is_set()` back-to-back rely on synchronous value-setting.
  - `EventBroadcaster` subscribers store `(queue, loop)` pairs and `publish()` always uses `loop.call_soon_threadsafe(queue.put_nowait, event)` — it's safe even when the publisher and consumer share a loop (slight deferral to next loop iteration is fine for events) and essential when they don't. The `_subs` dict is guarded by a `threading.Lock` so a WS disconnect-during-publish doesn't race the subs-list iteration.
  - Starlette's `TestClient.websocket_connect()` opens a NEW portal-thread with its own event loop. So the WS handler and the run task run on DIFFERENT loops even though they share the same `RunManager` singleton. This is the concrete reason the cross-loop-safe publish matters — without it, `put_nowait` on a queue bound to loop W from loop B silently drops events or schedules callbacks on the wrong loop.
  - The API router falls back to a global `RunManager()` (lazy-initialized on `request.app.state.run_manager`) if the gateway's lifespan didn't populate one. Tests that care about isolation construct their own manager and assign `app.state.run_manager` before `app.include_router(router)` — otherwise the first request creates a default manager that spans the whole process.
  - For the WS happy-path test, inner POST/confirm WORKS inside `with client.websocket_connect() as ws:` because the confirm endpoint doesn't spawn background work — it's a simple `submit_decision` that returns immediately. So sub-client operations are fine as long as they don't rely on post-return background tasks.
